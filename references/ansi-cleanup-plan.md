# ANSI 清理方案

## 目标

为 Codex PTY 和 Kimi CLI 的输出提供统一、低成本、可复用的 ANSI 转义码清理方案，避免颜色码、光标控制码、清屏码污染后续解析。

本文件只给方案，不在当前版本直接落脚本。

## 现状判断

本地环境可直接使用的通用工具：

- `perl`
- `sed`
- `awk`
- `tr`
- `col`
- `script`
- `python3`

本地未确认存在的专用工具：

- `ansi2txt`
- `ansifilter`

结论：

- 不应依赖专用第三方工具。
- MVP 采用系统已有工具组合更稳。

## 目标输出

无论来源是 Codex PTY 还是 Kimi CLI，都应尽量得到：

- 去颜色码
- 去光标移动码
- 去清屏码
- 保留普通换行和正文顺序
- 对常见退格覆写做基础清理

## 推荐方案

采用“两段式清理管道”：

1. 先用正则剥离 ANSI escape sequences
2. 再用 `col -b` 清理退格和控制性覆盖字符

推荐基线管道：

```bash
some_command 2>&1 \
  | perl -pe 's/\e\\[[0-9;?]*[ -\\/]*[@-~]//g; s/\e\\][^\\a]*(\\a|\\e\\\\)//g; s/\r(?!=\n)//g' \
  | col -b
```

说明：

- 第一段清理 CSI 类转义序列，如颜色、清屏、光标移动。
- 第二段补清理 terminal 覆写痕迹。
- `2>&1` 保证标准错误也被统一清洗。

## 为什么优先选 `perl + col -b`

- `perl` 对复杂 ANSI 模式比纯 `sed` 更稳。
- `col -b` 对退格类字符有现成处理。
- 两者都已在多数类 Unix 环境可用。
- 对长输出是流式处理，不需要整段加载到内存。

## 适配 Codex PTY

Codex PTY 常见问题：

- 带颜色的终端输出
- 进度刷新导致的回车覆写
- 局部清屏或光标移动

建议：

- 采集 PTY 输出后，统一经过清理管道再送入后续解析。
- 不要指望上游完全关闭颜色；即使设置了 `NO_COLOR=1`，也保留兜底清洗。

如果后续允许做工程实现，优先策略应是：

1. 先尝试关闭颜色输出
2. 再做 ANSI 清洗兜底

## 适配 Kimi CLI

Kimi CLI 常见问题：

- 彩色标题或状态提示
- spinner / 覆写行
- 错误输出中夹带 ANSI

建议：

- 所有 `kimi` 输出统一走同一条清理管道
- 对失败日志和正常输出一视同仁处理

## 轻量封装建议

后续如果要落地，建议不是为每个 agent 各写一套，而是做一个通用包装器，例如：

- shell function：`clean_ansi`
- 通用脚本：`scripts/clean-ansi.sh`
- 或统一日志采集函数：`capture_clean_output`

输入约定：

- 接收标准输入

输出约定：

- 输出已清洗纯文本

这样 Codex PTY 和 Kimi CLI 都只需要把原始流 pipe 进去。

## 备选方案

### 方案 A：上游禁色 + 下游兜底

可尝试给上游注入：

- `NO_COLOR=1`
- `TERM=dumb`

优点：

- 从源头减少污染

缺点：

- 不能覆盖所有工具
- 有些 CLI 仍会输出控制码

结论：

- 可以作为辅助手段，不能替代清理管道

### 方案 B：纯 `sed`

示意：

```bash
some_command 2>&1 | sed -E 's/\x1B\[[0-9;?]*[ -\/]*[@-~]//g'
```

问题：

- 对 OSC 等复杂序列覆盖不完整
- 对退格和覆写处理较弱

结论：

- 可作为极简兜底，不建议作为主方案

### 方案 C：Python 脚本

优点：

- 可维护性更高
- 更适合后续增强

缺点：

- 相比 `perl` 启动更重
- 当前只出方案，没必要先引入脚本文件

结论：

- 后续若要做复杂日志结构化，再升级为 Python 版本

## 性能判断

MVP 预期不会成为瓶颈，原因：

- 处理方式是流式的
- 会议输出量通常远小于构建日志
- `perl` 和 `col` 都是成熟命令行工具

注意：

- 不建议先把超长文本读入变量再一次性处理
- 应优先使用管道边读边清

## 接入点建议

建议只在两个位置接入，不要到处散落：

1. agent 原始输出采集层
2. 会议记录归档前的标准化层

这样做的好处：

- 采集后内容干净，便于解析
- 归档前再兜底一次，避免漏网控制码

## 验收标准

方案落地后应满足：

- 同一段 Codex PTY 输出清洗前后语义不变
- 同一段 Kimi CLI 输出清洗后无明显 ANSI 残留
- 解析共识、分歧、问题时不再被控制符干扰
- 普通文本吞吐不出现明显延迟

## 一句话结论

MVP 直接采用“上游尽量禁色 + 下游 `perl` 去 ANSI + `col -b` 去覆写”的统一管道，既通用，也足够稳。
