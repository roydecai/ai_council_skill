# AI Council Workflow

## 目标

本文件定义 AI Council 的完整执行工作流，重点补齐主持人侧的编排层，解决以下问题：

- 每轮如何开始、收集、结束
- agent 输出如何持久化
- `state.json` 何时更新、更新哪些字段
- 超时、缺席、降级如何落到实际动作
- 会议结束后如何产出决议、归档与清理

本文件与以下文档联动：

- 状态对象字段与恢复方式：`state-protocol.md`
- 各 agent 调用约束与踩坑：`agent-config.md`
- 降级规则：`degradation.md`
- ANSI 清理策略：`ansi-cleanup-plan.md`

## 一、总体编排原则

- 主持人是唯一编排者，也是唯一正式写入状态对象的人。
- 所有轮次都必须先落盘，再总结，再更新状态。
- 第 1 轮必须独立发言，第 2 轮起才交换观点。
- 每轮都必须形成 3 份产物：
  - agent 原始输出
  - 清洗后的轮次文件
  - 主持人轮次总结
- 会议过程中不得只依赖 PTY 滚屏或 session 历史，必须把关键内容写入工作目录。

## 二、会议目录初始化

目标：建立本次会议的最小可运行工作区，保证后续调用、持久化、归档和恢复都有统一落点。

推荐目录结构：

```text
<meeting-dir>/
├── .ai-council/
│   ├── state.json
│   ├── meeting-prompt.md
│   ├── orchestration-log.md
│   └── raw/
│       ├── round-1-gpt-codex.raw.log
│       └── round-1-kimi.raw.log
├── rounds/
│   ├── round-1-gpt-codex.md
│   ├── round-1-kimi.md
│   ├── round-1-pipi.md
│   └── round-1-summary.md
├── decisions/
│   ├── final-resolution.md
│   └── executive-summary.md
└── archives/
```

初始化步骤：

1. 创建临时工作目录，例如 `mktemp -d /tmp/ai-council-XXXXXX`
2. 进入目录并执行 `git init`
3. 创建 `.ai-council/`、`.ai-council/raw/`、`rounds/`、`decisions/`
4. 写入 `meeting-prompt.md`
5. 初始化 `.ai-council/state.json`
6. 把主持人准备过程简要记录到 `.ai-council/orchestration-log.md`

可直接执行的初始化示例：

```bash
MEETING_DIR=$(mktemp -d /tmp/ai-council-XXXXXX) && cd "$MEETING_DIR" && git init
mkdir -p .ai-council/raw rounds decisions archives
touch .ai-council/meeting-prompt.md .ai-council/orchestration-log.md .ai-council/state.json
```

初始化完成后，主持人至少要确认：

- 当前目录是 git repo
- `state.json` 已存在且可读
- 参会 agent 已确定
- 会议模式已确定
- 第 1 轮问题已写入材料包

## 三、阶段 1：需求确认

目标：明确会议主题、边界、产出物、复杂度、会议模式和参会阵容。

执行要点：

- 要求用户说明会议主题。
- 要求用户说明讨论边界，明确哪些内容在讨论范围内，哪些内容不在。
- 要求用户说明重点关注项，例如风险、技术方案、排期、资源、争议点。
- 判断会议类型：
  - 涉及具体程序、实现、架构、编码、调试、重构时，归为项目制会议。
  - 其他偏策略、分析、判断、创意、方案评估时，归为一般议题会议。
- 判断复杂度与会议模式：
  - 简单：边界清晰、资料少、主要做快速决策，进入 `light` 模式。
  - 标准：需要多方取舍与交叉回应，进入 `standard` 模式。
  - 深度：高风险、高不确定性或预计争议较大，进入 `deep` 模式。
- 明确参会智能体名单，并说明增减理由。
- 保证 MVP 至少 3 人：皮皮 + Kimi + GPT Codex。

阶段产出：

- 会议主题
- 会议边界
- 会议类型
- 复杂度等级
- 会议模式
- 参会名单
- 预期产出清单

## 四、阶段 2：资料准备

目标：由主持人整理可供多智能体引用的结构化材料包，并写入工作目录。

执行要点：

- 收集与主题相关的代码、文档、图片、音频、历史结论、会议记录。
- 将资料整理为“背景信息 / 当前约束 / 已知事实 / 待验证假设 / 关键文件”结构。
- 对项目制会议，优先补齐：
  - 仓库路径
  - 关键文件
  - 当前报错
  - 已知限制
  - 交付目标
- 对一般议题会议，优先补齐：
  - 背景上下文
  - 决策目标
  - 评价标准
  - 关键分歧
- 将结构化内容写入 `.ai-council/meeting-prompt.md`
- 同步初始化 `.ai-council/state.json`

推荐材料包结构：

```md
# 会议材料包

## 主题
## 背景
## 已知事实
## 当前约束
## 待验证假设
## 关键资料
- 文件 / 文档 / 图片 / 音频
## 本次希望达成的结果
## 第 1 轮问题
```

状态初始化至少填入：

- 当前模式
- 当前轮次
- 参会成员
- 初始待答问题
- 初始争议度
- 下一步动作
- 工作目录中的关键文件路径

## 五、阶段 3：会前拉起 agent

目标：在正式开始第 1 轮前，把所有参会 agent 拉起到可交互状态。

执行要点：

- Kimi：创建或指定固定 `session-id`
- Codex：在会议工作目录中启动 PTY 会话
- 记录每个 agent 的会话信息到状态对象：
  - `session_id`
  - `transport`
  - `status`
  - `last_seen_at`
- 如使用后台进程，记录 PID，供结束时清理

主持人检查项：

- Codex 已通过首次信任确认
- Kimi session 可复用
- 所有 agent 都知道三段式输出要求
- 所有 agent 都明确当前是第几轮

## 六、阶段 4：轮次执行编排

目标：把“开始一轮讨论”拆成主持人可重复执行的固定流程。

### 4.1 轮次硬规则

- `light` 模式固定 2 轮。
- `standard` 模式固定 3 轮。
- `deep` 模式最少 3 轮，最多 5 轮。
- 每轮每个活跃 agent 必须发言。
- 第 1 轮必须独立输出，不发送其他 agent 的答案。
- 第 2 轮起必须带入“其他观点摘要 + 本轮必须回答的问题”。
- 每轮结束后必须先收集输出，再更新状态，再决定是否进入下一轮。

### 4.2 单轮标准流程

每一轮都按以下顺序执行，不跳步：

1. 生成本轮任务
2. 广播给各 agent
3. 等待输出
4. 收集并清洗输出
5. 写入轮次文件
6. 主持人整理总结
7. 更新 `state.json`
8. 判断继续、补证、降级还是收敛

### 4.3 生成本轮任务

第 1 轮任务只包含：

- 会议材料包
- 当前问题
- 输出格式要求
- 独立思考要求

第 2 轮及以后必须追加：

- 上一轮总结
- 其他 agent 的观点摘要
- 当前仍开放的分歧
- 本轮必须回答的问题
- 是否存在缺席、超时、降级

推荐主持人任务模板：

```md
你正在参加 AI Council 第 N 轮讨论。

本轮目标：
- ...

你必须输出：
1. 观点
2. 思考逻辑
3. 支持材料

第 2 轮起额外输出：
4. 对其他观点的回应
5. 仍保留的分歧 / 异议 / 未决项
```

### 4.4 等待策略与超时阈值

MVP 推荐超时阈值：

- Kimi：`120s`
- Codex：
  - 非搜索型议题：`300s`
  - 联网搜索密集型议题：`600s`

等待原则：

- 先并行等待全部 agent
- 达到单 agent 阈值后，先标记 `timeout`
- 允许一次轻量重试
- 重试仍失败时，转 `degradation.md` 处理

### 4.5 输出采集与清洗

这是编排层必须显式做的动作，不能只看终端内容。

采集规则：

- 每个 agent 的原始输出先写入 `.ai-council/raw/`
- 原始输出保留原貌，便于排查
- 清洗后正文写入 `rounds/`
- 清洗动作应在采集层或归档前统一执行

推荐文件：

- 原始日志：`.ai-council/raw/round-<N>-<agent-id>.raw.log`
- 清洗正文：`rounds/round-<N>-<agent-id>.md`
- 主持总结：`rounds/round-<N>-summary.md`

清洗规则：

- Codex PTY 输出不得直接在启动命令后接管道
- PTY 输出采集后，再按 `ansi-cleanup-plan.md` 清洗
- Kimi 若使用非 PTY 调用，可在采集时直接得到较干净文本
- 无论来源如何，归档前都应再做一次兜底清洗

### 4.6 主持人轮次总结

每轮结束后，主持人必须生成 `rounds/round-<N>-summary.md`，至少覆盖：

- 每个 agent 的核心观点
- 每个 agent 的逻辑链
- 每个 agent 的支持材料
- 本轮新增共识
- 本轮新增分歧
- 缺席或失败成员
- 下轮必须回答的问题
- 主持人对争议度的判断

推荐格式：

```md
## 第 N 轮总结

### Agent A
- 观点：
- 逻辑：
- 支持材料：

### Agent B
- 观点：
- 逻辑：
- 支持材料：

### 本轮共识

### 本轮分歧

### 本轮缺席 / 异常

### 下轮必须回答的问题
```

### 4.7 状态更新

每轮结束后，主持人必须立刻更新 `.ai-council/state.json`，不能拖到会议结束。

至少更新以下内容：

- `meeting.updated_at`
- `round.current`
- `round.history`
- `positions`
- `consensus`
- `disagreements`
- `open_questions`
- `controversy`
- `artifacts`
- `next_step`

状态更新的最小要求：

- 本轮谁发言了，谁没发言
- 每个 agent 当前立场是什么
- 本轮新达成了什么共识
- 本轮新暴露了什么分歧
- 下一轮是否继续

### 4.8 第 2 轮起的观点交换机制

第 1 轮结束后，主持人要做一次显式汇总，再进入第 2 轮。最少包括：

- Kimi 的核心观点摘要
- Codex 的核心观点摘要
- 主持人的中间判断
- 明确冲突点
- 需要互相回应的点

主持人不应把整段原文原样转发给所有 agent，而应压缩成：

- 他人观点摘要
- 已出现的证据
- 尚未覆盖的反证
- 本轮必须回应的具体问题

### 4.9 轮次结束后的决策分支

每轮结束后只允许进入以下四种分支之一：

- `start_next_round`
- `request_evidence`
- `degrade_and_continue`
- `finalize`

若状态对象、轮次总结和落盘文件三者不一致，不得进入下一轮。

## 七、阶段 5：收敛判断

目标：在讨论充分后判断是否已经形成稳定结论。

判断标准：

- 关键问题是否已有明确方案。
- 高重要性分歧是否已经解决。
- 未解决问题是否已缩小到可接受范围。
- 方案是否具备执行条件。
- 所谓“趋同”是否有新增证据支撑，而不是单纯附和。

处理规则：

- `light` 模式第 2 轮后判断是否收敛。
- `standard` 模式第 3 轮后判断是否收敛。
- `deep` 模式第 3 轮后若关键问题已稳定，进入阶段 6。
- `deep` 模式第 3 轮后若仍有明显分歧，继续第 4 轮。
- `deep` 模式第 4 轮后仍有高重要性分歧，继续第 5 轮。
- `deep` 模式第 5 轮后仍未收敛，记录未收敛事项并进入阶段 6。

主持人额外检查：

- 本轮是否出现新增实质性分歧。
- 争议度是下降，还是只是被措辞掩盖。
- 是否存在“没人反对，但也没人补证据”的伪收敛。

## 八、阶段 6：决议输出

目标：形成完整、忠实、可复核的会议结论。

必须输出：

- `decisions/final-resolution.md`
- `decisions/executive-summary.md`

分歧处理规则：

- 高重要性分歧：在完整决议和决议摘要中都体现。
- 低重要性分歧：仅在完整决议中体现。

完整决议必须覆盖：

- 讨论背景
- 参会者
- 会议模式
- 各轮核心观点
- 共识内容
- 分歧内容
- 最终决策或待定项
- 建议后续动作
- 若发生异常降级，附降级说明

## 九、阶段 7：归档同步

目标：将会议结果整理为长期可检索资产。

执行要点：

- 项目制会议归档到项目文件夹。
- 非项目会议按上下文选择合适位置。
- 命名参考：`YYYYMMDD_council_主题.md`
- 命名风格遵循 Obsidian 文档治理规范。
- 同步到 Obsidian，并保留可分享链接或引用路径。

建议归档内容：

- `meeting-prompt.md`
- `state.json`
- `rounds/` 下全部轮次正文
- `decisions/` 下决议与摘要
- 原始日志中有必要保留的关键片段

## 十、阶段 8：资源清理

目标：在保证产物归档完成后，释放会话、临时目录和后台进程。

执行顺序：

1. 确认 `state.json` 已更新为最终状态
2. 确认 `decisions/` 已生成
3. 归档 `state.json`、`rounds/`、`decisions/`
4. 杀掉本次会议启动的后台进程
5. 清理临时工作目录
6. 记录清理完成时间

注意：

- 必须先归档，再删除
- 不得先删工作目录再找回产物
- Kimi 若使用持久 session，可不删 session，只在状态对象中标记保留

## 十一、用户触达规则

- Telegram 或飞书只发送摘要和链接。
- 不直接发送全文内容。
- 若需要全文，提供 Obsidian 链接或归档路径。

## 十二、主持人一句话执行模板

先建工作区和状态，再逐轮“发题 - 等待 - 收集 - 清洗 - 落盘 - 总结 - 更新状态”，最后输出决议、归档并清理。
